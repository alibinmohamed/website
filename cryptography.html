MODULE 01 · WHY

What is cryptography, and why?

Cryptography is the art of turning data you don't want anyone to read into data they can't read — and back again. It exists to do four jobs.

GoalPlain English
ConfidentialityOnly the right person can read it.
IntegrityThe message hasn't been changed in transit.
AuthenticationThe message really came from who it claims.
Non-repudiationThe sender can't later deny sending it.

Every algorithm in the rest of this course is just a way to achieve one or more of those four goals. We'll meet them one at a time.

Encoding ≠ encryption. Base64, hex, URL-encoding are all reversible by anyone. They hide nothing. The lab below has you experience that the hard way.

Now that we know why, the next chapter answers how did people do this before computers?

MODULE 02 · CLASSIC CIPHERS

The first ciphers — substitution

Before computers, people encrypted by hand. The simplest method is to replace each letter with a different letter. That's a substitution cipher.

CipherRuleExample: HELLO →
Caesar (≈50 BC)Shift every letter by 3 (A→D, B→E…)KHOOR
ROT13Caesar with shift 13 — its own inverseURYYB
AtbashFlip the alphabet (A↔Z, B↔Y…)SVOOL

These are easy to break. Caesar has only 25 possible shifts; you can try them all in seconds. And in English, the most common letter is almost always e. That trick is called frequency analysis, and it's the lab below.

Single-letter substitution falls fast. Chapter 3 asks: can we make substitution stronger?

MODULE 03 · STRONGER SUBSTITUTION

Vigenère — substitution with a keyword

Caesar shifts every letter by the same amount. Vigenère (1500s) shifts by a keyword: each letter of the key gives a different shift, and the key repeats for as long as the message.

Plain: HELLOWORLD Key: KEYKEYKEYK Cipher: RIJVSUYVJN

For 300 years it was nicknamed “the unbreakable cipher.” Then in the 1860s Charles Babbage and Friedrich Kasiski showed that frequency analysis still works — you just have to find the key length first.

The lab below uses a tiny vig helper inside the sandbox so you can try encrypting and decrypting in seconds.

The lesson: any cipher humans can do by hand eventually falls. We need maths and computers. Chapter 4.

MODULE 04 · SYMMETRIC

Symmetric encryption — one shared key

Modern symmetric encryption uses a single key that both sides share. Same key encrypts and decrypts.

AlgorithmYearStatus
DES1977Broken since 1999. Don't use.
3DES1998DES three times. Slow, retired.
AES2001Today's standard. 128/192/256-bit keys.
ChaCha202008AES alternative, fast on phones.

In the lab you'll use openssl enc -aes-256-cbc to encrypt a file, then decrypt it back. The same key (here, a password) is used for both. That single shared key is exactly what makes symmetric encryption fast — and what makes the next question hard.

What it achieves & what it can't

Objective

Confidentiality. Hide the message so only someone holding the same secret key K can read it. Fast, unbroken at AES-256 — perfect for big files and live streams.

Weakness

Key distribution. Both sides need the same K before they can talk. If you send K through the same channel an attacker is watching, they grab K too and decrypt everything. (See diagram below.)

Alternatives

Diffie-Hellman derives K in public without ever sending it. Asymmetric encryption (Chapter 7) lets Alice encrypt K with Bob's public key. Or deliver K out-of-band (in person, QR code, hardware token).
If the key travels through the same channel as the messages, anyone listening (Eve) can capture it and decrypt everything.

AES only encrypts 16 bytes at a time. To encrypt a long file we chain blocks together. How we chain them turns out to matter a lot. Chapter 5.

MODULE 05 · MODES

Modes of operation — chaining the blocks

Block ciphers like AES encrypt 16 bytes at a time. To encrypt a long message we string blocks together using a mode. The choice matters.

  • ECB — each block encrypted independently. Identical plaintext blocks produce identical ciphertext blocks. Patterns leak through. Don't use it. The famous “ECB Penguin” image is the standard demo.
  • CBC — each block is XORed with the previous ciphertext block before encrypting. Patterns disappear.
  • GCM — modern authenticated mode. Detects tampering for free. The default in TLS 1.3.

In the lab you'll encrypt the same file with ECB and with CBC and compare the bytes. CBC's output looks like random noise; ECB's gives away its structure.

We have hidden the data. But how do we know nobody changed it? That's what hashing is for. Chapter 6.

MODULE 06 · INTEGRITY

Hashing — fingerprints for data

A hash function turns any input into a fixed-size fingerprint. Same input always gives the same hash; change one bit and the hash changes completely.

FunctionOutputStatus
MD5128 bitsBroken — collisions easy
SHA-1160 bitsBroken since 2017
SHA-256256 bitsModern default
SHA-3, BLAKE2/3variableModern alternatives

Hashes are used for: file integrity (compare sums), password storage (with bcrypt / Argon2 to slow attackers down), and digital signatures (Chapter 8). Add a secret key to a hash and you get an HMAC: anyone with the key can verify the message; nobody without it can forge one.

What it achieves & what it can't

Objective

Integrity. Detect any change to the data. One bit flipped → a totally different fingerprint. Cheap to compute, easy to compare.

Weakness

No identity, no secrecy. An attacker who can edit the data can also re-hash it and replace both. A hash alone proves nothing about who sent it. Old hashes (MD5, SHA-1) also have collision attacks.

Alternatives

HMAC mixes a shared key into the hash so attackers can't forge it. Digital signatures (Chapter 8) bind the hash to a private key. Use SHA-256 or SHA-3 instead of MD5/SHA-1.

So far everything we did needed both sides to share a key in advance. Chapter 7 solves how two strangers share a key in the first place.

MODULE 07 · ASYMMETRIC

Asymmetric encryption — two keys, one pair

The breakthrough of the 1970s: two keys that are mathematically linked. A public key everyone can see, and a private key only you have.

If Alice encrypts with Bob's public key, only Bob's private key can decrypt it. They never had to share a secret first. The most famous algorithm is RSA (Rivest, Shamir, Adleman, 1977). Modern alternatives include ECC (Elliptic Curves) and post-quantum lattice schemes.

Asymmetric is slow, so in real life we use it to exchange a short symmetric key, then continue with AES. That's how TLS, SSH and PGP all work.

What it achieves & what it can't

Objective

Confidentiality without a shared secret. Anyone can encrypt to Bob using his public key; only Bob's private key can decrypt. Solves Chapter 4's key-distribution problem.

Weakness

Slow + identity trust. RSA is ~1000× slower than AES, so we never encrypt big files with it directly. Worse, if Eve tricks Bob into using her public key as Alice's, she reads everything (man-in-the-middle).

Alternatives

Hybrid encryption: encrypt the data with AES, encrypt only the AES key with RSA. Use certificates (PKI) so a CA vouches that a public key really belongs to its owner. ECC gives the same security with smaller keys; post-quantum schemes are coming.
Eve still listens, but the public key is harmless on its own — only Bob's private key can decrypt.

We've used the public key to send a secret. What if we flip it and use the private key to prove something? That's a digital signature. Chapter 8.

MODULE 08 · SIGNATURES

Digital signatures — sign with private, verify with public

Reverse the keys. If you sign data with your private key, anyone can verify with your public key that only you could have signed it. That's a digital signature.

In practice we sign a hash of the message rather than the whole message — much faster, and good enough thanks to Chapter 6's integrity guarantee. Combine signatures with a chain of trust (X.509 certificates issued by a CA) and you get HTTPS, signed software updates, and signed email.

Bob receives: message + signature 1. Hash the message himself. 2. Decrypt the signature with Alice's public key — gives a hash. 3. If the two hashes match, the message really came from Alice AND has not been changed.

What it achieves & what it can't

Objective

Authenticity + integrity + non-repudiation. Prove who signed what, and stop the signer denying it later. Used for HTTPS certificates, signed software updates, signed email.

Weakness

Trust in the public key. Verification only works if Bob has the real Alice's public key. If Eve planted hers, she signs as “Alice” and Bob believes it. Private-key theft is also game-over — anyone with priv_A can impersonate Alice.

Alternatives / fixes

Certificates & PKI (X.509) so a CA vouches for the public key. Web of trust (PGP) where friends sign each other's keys. Hardware tokens (YubiKey, TPM) keep the private key off disk so it can't be copied.

Where does all this show up in real life? Chapter 9.

MODULE 09 · IN THE WILD

Crypto in the wild

TLS, SSH, PGP, signed software updates, password managers, blockchains — every one of them is a recipe built from the ingredients in chapters 4 to 8.

A typical TLS handshake:

  1. Asymmetric (RSA / ECDH from Chapter 7) to exchange a fresh symmetric key.
  2. Symmetric (AES-GCM from Chapters 4 and 5) for the actual data.
  3. Hash + signature (Chapters 6 and 8) so both sides know nothing was tampered with and to prove the server's identity.

In the lab you'll inspect a real TLS certificate, verify a file's checksum, and confirm gpg is installed — ready for the capstone.

Capstone — your tiny PGP

Use everything in one go. In the sandbox, build a “send Bob a secret note” pipeline:

  • Bob: openssl genrsa + openssl rsa -pubout — gives you a key pair (Chapter 7).
  • Alice: write a message, encrypt it with AES (Chapter 4).
  • Alice: encrypt the AES password with Bob's public key (Chapter 7).
  • Alice: sign the AES file with her own private key (Chapter 8).
  • Bob: verify Alice's signature, decrypt the AES password with his private key, then decrypt the message with that password.

Congratulations — you've just built, by hand, the same recipe that PGP and every TLS connection on the planet use.
