MODULE 01 · FOUNDATIONS

Introduction to the Web

The web is a layer that runs on top of the Internet. The Internet is just wires, radios and routers moving raw data around. The web is what happens when one computer (a client) asks another computer (a server) for a document and that document is rendered as a page you can read, click and type into.

Web vs Internet

  • Internet — the global network of networks. Carries every kind of traffic: web, email, video calls, gaming, file transfer.
  • Web — one specific service that uses the Internet, defined by three things invented at CERN in 1989: URLs (addresses for documents), HTTP (the request language), and HTML (the document format).

The Three Building Blocks of Every Web Page

LanguageJobYou'll see it as…
HTMLStructure / contentHeadings, paragraphs, links, forms
CSSLook & feelColours, fonts, layout, spacing
JavaScriptBehaviourInteractive widgets, fetching data without reloading

Why Beginners Should Learn This Course

  • Every modern app talks HTTP somewhere. Mobile apps, games, banking, IoT — they all speak the same protocol.
  • You cannot break, defend, or build something well if you don't know how it works.
  • The vocabulary in this course (URL, header, cookie, proxy, DNS) is shared across Web Security, Penetration Testing, Cloud Security and Application Security tracks.
Professor's note: when people say "web app" they usually mean something you use through a browser. When they say "API" they usually mean the same thing without the visual layer. The protocol underneath is identical.

MODULE 02 · WHO TALKS TO WHOM

Clients & Servers

Every web interaction is a request from a client and a response from a server. There are no surprises in this model — once you internalise it, every confusing thing you'll encounter later (caches, proxies, CDNs, load balancers) is just a box that sits between the two.

The Client

  • Initiates the conversation.
  • Examples: a browser (Chrome, Firefox), a mobile app, the curl command, another server calling an API.
  • Speaks the same HTTP either way — the server can't tell a browser from curl unless you tell it (via the User-Agent header).

The Server

  • Listens on a port (typically 80 for HTTP, 443 for HTTPS).
  • Returns a response: a status code, headers, and (usually) a body.
  • Common server software: nginx, Apache, IIS, plus application-server frameworks like Express, Django, Flask, Spring.

The HTTP Request, in Plain English

GET /index.html HTTP/1.1 Host: example.com User-Agent: Mozilla/5.0 Accept: text/html (blank line)

The first line is the request line (method, path, version). After it come headers, then a blank line, then an optional body (used by methods like POST and PUT).

The HTTP Response

HTTP/1.1 200 OK Server: nginx Content-Type: text/html Content-Length: 1270 (blank line) <!doctype html><html><body><h1>It works!</h1></body></html>

HTTP Methods You'll Actually See

MethodPurposeBody?
GETRead a resourceNo
POSTCreate / submit a formYes
PUTReplace a resourceYes
PATCHPartially update a resourceYes
DELETERemove a resourceOptional
HEADLike GET but only the headersNo
OPTIONSAsk the server what it allowsNo
Lab 2 — See the Conversation Live
  1. Watch a full request and response: curl -v http://example.com.
  2. Send a POST: curl -X POST http://example.com.
  3. Open a raw TCP connection and pretend to be a browser: nc example.com 80.
  4. Run your own tiny server in this sandbox: python3 -m http.server 8000.
Deliverable: notice how every request begins with a method + path line, and every response begins with a status code. That is the entire web in two lines.
MODULE 03 · PROTOCOLS

HTTP vs HTTPS

HTTP sends every byte of the conversation in plain text — anyone on the network path can read it. HTTPS is exactly the same HTTP, wrapped inside an encrypted tunnel called TLS. That single change is what makes online banking, login forms and private messaging viable.

The Two Differences You Care About

HTTPHTTPS
Default port80443
Encrypted?NoYes (TLS)
Identity proven?NoYes — server presents an X.509 certificate
Tamper-evident?NoYes — TLS detects any in-flight modification
Used in production?Almost never anymoreThe default everywhere

What TLS Actually Does

  1. Identity: the server presents a certificate signed by a Certificate Authority (CA) that the client already trusts.
  2. Key exchange: client and server agree on a fresh, single-use secret key without ever putting it on the wire.
  3. Encryption: from then on, every byte is encrypted with that key.

Once TLS finishes, everything inside the tunnel is regular HTTP. That is why curl http://... and curl https://... look almost identical from a tooling perspective — only the transport changed.

Security angle: a "valid" HTTPS connection only proves that you are talking to the certificate's owner — it does not prove the page is honest. Phishing sites can and do use real HTTPS certificates.

HTTP Status Codes — The Tens You Must Know

RangeMeaningExamples
1xxInformational100 Continue
2xxSuccess200 OK · 201 Created · 204 No Content
3xxRedirect301 Permanent · 302 Found · 304 Not Modified
4xxClient error400 Bad Request · 401 Unauthorized · 403 Forbidden · 404 Not Found
5xxServer error500 Internal · 502 Bad Gateway · 503 Service Unavailable
Lab 3 — Watch HTTP Become HTTPS
  1. Plain HTTP: curl http://example.com.
  2. Same site over TLS: curl https://example.com.
  3. Peek inside the TLS handshake: openssl s_client -connect example.com:443.
  4. Headers only: curl -I https://example.com.
Deliverable: a one-paragraph explanation of why a Wi-Fi attacker on the same coffee-shop network can read the URL when you visit http://example.com but not when you visit https://example.com.
MODULE 04 · NAMES

DNS, Domains & Subdomains

Computers route packets by IP address, not by name. DNS — the Domain Name System — is the global directory that turns names humans can remember (utb.edu.bh) into addresses computers can use (185.220.101.4).

The Domain Hierarchy, Right-to-Left

Read a domain right-to-left. Each dot is a level in a tree:

www . cyber . utb . edu . bh . ↑ ↑ ↑ ↑ ↑ ↑ | | | | | └─ root (always implied) | | | | └────── TLD ("bh" — Bahrain) | | | └─────────── second-level ("edu") | | └──────────────── registered domain ("utb") | └──────────────────────── subdomain ("cyber") └─────────────────────────────── host / sub-subdomain ("www")

The DNS Records You'll Actually Look Up

TypeMeansExample value
AIPv4 address93.184.216.34
AAAAIPv6 address2606:2800:220:1:248:1893:25c8:1946
CNAME"This name is an alias for that name"www → example.com
MXMail exchanger for the domain10 mail.example.com
NSAuthoritative nameservera.iana-servers.net
TXTFree-form text (SPF, DKIM, verification)v=spf1 -all

How a DNS Lookup Actually Happens

  1. Your OS asks a resolver (often your router or 1.1.1.1).
  2. The resolver asks a root server for the TLD's nameservers.
  3. It then asks the TLD nameserver for the domain's nameservers.
  4. Finally it asks the domain's authoritative nameserver for the record. The answer is cached for the record's TTL.

Subdomains Are Just Free Labels

The owner of example.com can create as many subdomains as they want — blog.example.com, api.example.com, shop.example.com. Each subdomain is technically a different host and can point to a completely different server.

Lab 4 — Resolve Like a Sysadmin
  1. A record: dig example.com.
  2. Mail servers: dig -t MX example.com.
  3. Nameservers: dig -t NS example.com.
  4. Registration info: whois example.com.
Deliverable: a short table listing the A, MX and NS records you found and a sentence describing why the MX server is usually a different host from the web server.
MODULE 05 · ADDRESSES

URLs & Parameters

A URL is a structured string that tells a browser where a resource is and how to ask for it. Every part has a name and a job. Once you know all the pieces, you can read any link on the Internet.

Anatomy of a URL

https:// user:pass@ api.example.com: 8443 /v1/users ?id=42&sort=name #section-3
PartPurposeOften optional?
Scheme (https)Protocol to useRequired
Userinfo (user:pass)Basic-auth credentialsYes (rarely used today)
Host (api.example.com)Which server to talk toRequired
Port (8443)Which TCP portYes — defaults to 80/443
Path (/v1/users)Which resource on the serverDefaults to /
Query (?id=42&...)Named parameters for the serverYes
Fragment (#section-3)Position within the pageYes — never sent to the server

Query-string Parameters (a.k.a. "GET parameters")

A query string starts with ? and is a list of name=value pairs separated by &. Spaces and special characters must be URL-encoded: hello world becomes hello%20world or hello+world.

$ curl 'http://example.com/search?q=hello%20world&lang=en'

Form Data — POST Parameters

When a form uses method="POST", the parameters travel in the request body instead of the URL. They are still name=value pairs, but the body's Content-Type tells the server how to parse them (application/x-www-form-urlencoded or multipart/form-data).

$ curl --data 'name=ali&role=student' http://example.com/register
Security angle: never put passwords or session tokens in the query string — URLs end up in browser history, server logs and the Referer header. Use POST bodies or the Authorization header instead.
Lab 5 — Speak the URL Language Fluently
  1. Send query parameters: curl 'http://example.com/?q=hello&lang=en'.
  2. Submit form data with POST: curl --data 'name=ali&role=student' http://example.com.
  3. URL-encode safely: curl --data-urlencode 'q=hello world' http://example.com.
  4. Hit a non-default port: curl http://example.com:8080/.
Deliverable: annotate one of your own URLs (e.g. a Google search you ran today) and label every part using the table above.
MODULE 06 · METADATA

Headers, Cookies & Sessions

The body of an HTTP request or response is the obvious bit. The headers are where most of the interesting work happens — language preferences, authentication, caching policies, and (on the server side) the famous cookies that turn the stateless web into something that remembers who you are.

Common Request Headers

HeaderWhat it tells the server
HostWhich virtual host to serve (one IP can host many domains).
User-AgentWhich client/browser is sending the request.
AcceptWhich content types the client can read.
Accept-LanguagePreferred natural languages.
AuthorizationCredentials (Basic, Bearer, etc.).
CookieStored state from previous responses.

Common Response Headers

HeaderWhat the server is telling you
Content-TypeWhat the body is (text/html, application/json…).
Content-LengthSize of the body in bytes.
Cache-ControlHow long this response can be reused.
Set-Cookie"Please remember this and send it back next time."
LocationWhere to redirect to (with 3xx codes).
ServerWhat software is answering (often hidden in production).

Cookies — The Web's Memory

HTTP itself is stateless: the server forgets you the moment the response is sent. To keep you logged in, the server includes a Set-Cookie header. Your browser stores it, and includes it on every subsequent request to the same domain via the Cookie header.

Set-Cookie: session=abc123def; Path=/; HttpOnly; Secure; SameSite=Lax
  • HttpOnly — JavaScript cannot read the cookie. Defeats most XSS cookie-theft attacks.
  • Secure — only sent over HTTPS.
  • SameSite — controls when the cookie is sent on cross-site navigation (Lax / Strict / None). Mitigates CSRF.
Lab 6 — Inspect and Manipulate Headers
  1. List the response headers: curl -I https://example.com.
  2. Send a custom header: curl -H 'X-Demo: 1' http://example.com.
  3. Send a cookie: curl -b 'session=abc123' http://example.com.
  4. Save received cookies into a jar: curl -c cookies.txt http://example.com.
Deliverable: open cookies.txt in the sandbox (cat cookies.txt) and explain what each column means using the section above.
MODULE 07 · BOXES IN THE MIDDLE

Proxies & Caching

In real life there are almost always boxes between the client and the server. They relay, inspect, cache or rewrite traffic. The two most important ones are proxies (which forward your request) and caches (which remember the response so the next request is faster).

Forward Proxy vs Reverse Proxy

Forward ProxyReverse Proxy
Sits in front ofThe clientThe server
Used byCompanies, schools, security toolsWeb service operators
Typical purposeFilter/log outbound traffic, inspect HTTPS, anonymiseLoad balance, terminate TLS, cache, hide origin
ExamplesBurp Suite, OWASP ZAP, corporate proxynginx, Cloudflare, AWS ALB

Caching, Briefly

  • Browser cache — your own browser stores responses locally so returning to a page is instant.
  • CDN (Content Delivery Network) — Cloudflare, Akamai, Fastly. A global mesh of reverse proxies that caches popular content close to the user.
  • Cache-Control headers — the server tells caches how long, and under what conditions, the response can be reused.

Following Redirects

Servers often respond with 301 or 302 and a Location: header pointing somewhere else (e.g. HTTP → HTTPS, old URL → new URL). Browsers follow them silently. curl does not, unless you ask: -L / --location.

Lab 7 — Send Traffic Through a Proxy
  1. Start a tiny local web server in another part of the sandbox: python3 -m http.server 8080 (then Ctrl+C when done).
  2. Send a request through a proxy address: curl -x http://127.0.0.1:8080 http://example.com.
  3. Bypass the cache: curl -H 'Cache-Control: no-cache' http://example.com.
  4. Follow redirects: curl -L http://example.com.
Deliverable: a paragraph describing one real-world use case for a forward proxy (e.g. school filtering) and one for a reverse proxy (e.g. Cloudflare).
MODULE 08 · ARCHITECTURE

Web Application Architecture

Modern web applications are not single programs — they are usually several layers cooperating over HTTP. Knowing where the boundaries are makes everything you read in tutorials, blog posts and CTF challenges click into place.

The Three Tiers

  • Front-end — runs in the user's browser. HTML, CSS, JavaScript, often a framework like React or Vue. Talks to the back-end via HTTP.
  • Back-end — server-side code (Python/Flask, Node/Express, Java/Spring, Go, Ruby/Rails…). Receives HTTP requests and decides what to do.
  • Database — stores persistent state. Relational (PostgreSQL, MySQL, SQLite) or document (MongoDB, DynamoDB).

APIs & JSON

Most back-ends today expose a REST API that returns JSON instead of HTML. The same back-end can serve a website, a mobile app, and a CLI client simply by speaking JSON over HTTP.

{ "id": 583231, "login": "octocat", "name": "The Octocat", "company": "@github", "public_repos": 8 }

Common Stacks You'll Hear About

StackPieces
LAMPLinux + Apache + MySQL + PHP
MEAN / MERNMongo + Express + Angular/React + Node
JAMstackJavaScript + APIs + pre-rendered Markup
SPA + RESTSingle-Page App in the browser, JSON REST on the server
Professor's note: "GraphQL", "gRPC" and "WebSockets" are alternative protocols you'll meet next, but every one of them sits on top of (or alongside) HTTP. Master HTTP first.
Lab 8 — Talk to a Real API
  1. Hit a real REST API: curl https://api.github.com/users/octocat.
  2. Ask politely for JSON: curl -H 'Accept: application/json' https://api.github.com.
  3. Pretty-print it: curl https://api.github.com | jq ..
  4. Pretend to update something: curl -X PUT http://example.com/item/1.
Deliverable: a screenshot of the prettified JSON response and a one-sentence guess at which database table backs the octocat user record.
MODULE 09 · TOOLS OF THE TRADE

The Developer Toolbox

You now know the language; this last chapter teaches you the tools. Real practitioners live in three places: the browser DevTools, a command-line HTTP client (usually curl or HTTPie), and an interception proxy (Burp / ZAP).

Browser DevTools (F12 / Cmd+Opt+I)

  • Network tab — every request the page makes, with timing, headers, status, body. The single most useful tab on the entire web.
  • Console — the JavaScript REPL of the page. Type document.title and hit Enter.
  • Application / Storage — cookies, localStorage, sessionStorage, IndexedDB.
  • Elements — live DOM tree; you can edit it on the fly.

Command-Line Power Moves with curl

FlagWhat it does
-vVerbose — show request & response in detail.
-IHeaders only (HEAD).
-LFollow redirects.
-o fileSave response body to a file.
-sSilent — drop progress bars, pipe-friendly.
-wPrint stats after the request (%{time_total}).
-H 'X: y'Send a custom header.
-d k=vSend a POST body.
-x proxySend through a proxy (Burp / ZAP).

Pipes and One-Liners

curl plays nicely with classic Unix tools. The shell pipe (|) connects them.

$ curl -s https://api.github.com | jq . $ curl -s http://example.com | head $ curl -s http://example.com | grep -i <title>
Lab 9 — Be Your Own DevTools
  1. Verbose request: curl -v https://example.com.
  2. Save the page: curl -o page.html http://example.com.
  3. Pipe into a filter: curl -s http://example.com | head.
  4. Time the request: curl -w '%{time_total}\n' -o /dev/null -s http://example.com.
Deliverable: compare the timing you measured to the same request from a different region (e.g. ask a friend to run it). Networks aren't equal.

Capstone — Put It All Together

You've now covered every basic concept: clients and servers, HTTP vs HTTPS, DNS, domains and subdomains, URLs and parameters, headers and cookies, proxies and caching, web app architecture, and the developer toolbox. For your capstone, build a tiny "request explorer" using only what this course taught:

  • Pick any public website you use daily.
  • Use dig, whois, and host to map its DNS footprint (A, MX, NS, two interesting subdomains).
  • Use curl -v to capture one full request/response cycle and label every header.
  • Use curl -I to identify which caching strategy the site uses (Cache-Control, ETag, Last-Modified).
  • Use openssl s_client to extract the TLS certificate's CN, issuer and validity dates.
  • Write a one-page report describing what you found and which OWASP-style risks each response header does (or does not) defend against.

When your report is ready, share it in the club Discord for feedback from a senior member. Completing this capstone qualifies you to move on to Web Security proper, Application Security, and the OWASP Top 10.
