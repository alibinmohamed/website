MODULE 01 · ETHICS

Before you scan a single port

A penetration test is a controlled, written-down, agreed-with-the-client attack on systems someone else owns. Without that paperwork it is just a crime. The first chapter is the only one where you don't run any tools — but it is also the chapter that decides whether your career survives the next eleven.

The four documents every engagement needs

  • Statement of Work (SOW) — what you are being paid to do, by when, for how much.
  • Rules of Engagement (ROE) — exactly which targets, dates, hours, techniques are allowed and forbidden.
  • Authorization Letter — countersigned proof you can show to anyone (incident responders, ISP, police) that says “yes, this person is allowed to test these systems”.
  • Non-Disclosure Agreement (NDA) — binding promise that the secrets you find don't leave the engagement.
No box on the left signed? No green box on the right. The engagement does not start.
Legal landmine. Pen-testing without written authorization is a crime in nearly every jurisdiction. In the U.S. it is the Computer Fraud and Abuse Act; in the U.K. it is the Computer Misuse Act 1990; in Bahrain it is the Personal Data Protection Law (30/2018) plus Cybercrime Decree-Law 60/2014. Even a bare nmap scan of a third-party network can land you in court.

In the lab you'll read four pre-built documents inside the sandbox: ROE.md, AUTHORIZATION-LETTER.txt, LEGAL.md, and the file .target that names the only IP/hostname these labs are allowed to point at (scanme.nmap.org + localhost).

Now that the paperwork is signed, the next chapter answers: which way of testing did the client buy?

MODULE 02 · METHODOLOGY

White, grey, black — and which methodology you follow

Pen-tests come in three flavours depending on how much information the client gives you. Every engagement also follows a methodology — a checklist that tells reviewers, regulators and the next tester after you what was actually done.

Box types

The choice belongs to the client and is locked in the ROE — it is not a slider you can move during the test.

Methodologies you can name in your report

StandardUse
PTESPenetration Testing Execution Standard. 7 phases. The default external-engagement structure.
OWASP WSTGWeb Security Testing Guide — the gold standard for testing a web app.
NIST SP 800-115Technical Guide to Information Security Testing — used when the client must align with NIST 800-53 controls.
OSSTMMOpen Source Security Testing Methodology Manual — wider scope than PTES, common in audits.
MITRE ATT&CKAdversary-emulation reference. Useful for red-team / purple-team engagements that need to map findings to a known threat actor.

In the lab you'll read four pre-written cheat sheets inside the sandbox: the methodologies overview, the box-type comparison, the seven PTES phases, and the NIST SP 800-115 outline.

You know the rules and the playbook. Time to actually find out what the target looks like — and chapter 3 does it without touching them once.

MODULE 03 · OSINT

Passive reconnaissance — learning without touching

Before you send a single packet to the target, you can learn an embarrassing amount about them from public records. This is passive recon or OSINT (Open-Source Intelligence). The target's logs never show a thing, because every query goes to a third party.

The four passive-recon staples

ToolWhat it tells you
whois example.comRegistrar, registration date, contact emails, registered nameservers.
dig example.comA records — the IPv4 addresses behind the hostname.
dig MX example.comMail servers — useful for phishing scope and email vendor identification.
dig -x 8.8.8.8Reverse DNS — what hostname an IP claims to be.

Beyond DNS, OSINT also covers Google dorking (site:example.com filetype:pdf), Shodan (org:"Example Corp"), the Wayback Machine, the employee directory on LinkedIn and any data breaches the company has been involved in.

Passive recon ends when you've drawn the map. Chapter 4 is where you start touching the target — carefully.

MODULE 04 · NMAP

Active reconnaissance with nmap

nmap is the swiss-army-knife of active recon. In four commands you go from "is anything alive at this address?" to "that web server is running Apache 2.4.7 on Ubuntu, here are its open ports, here are the matching CVEs".

The four scan types you'll use the most

Walk left to right as the engagement matures: ping sweep first, aggressive last.
$ nmap -sn 10.0.0.0/24 # 1. host discovery $ sudo nmap -sS scanme.nmap.org # 2. SYN scan (raw socket -> sudo) $ nmap -sV scanme.nmap.org # 3. service versions $ nmap -A scanme.nmap.org # 4. OS + scripts + versions $ nmap --script vuln scanme.nmap.org # 5. NSE vulnerability checks
scanme.nmap.org is the only public host the nmap project explicitly authorises everyone to scan. Use it for these labs and don't point any of these commands at a system you don't own and haven't been written authorization for.

Open ports tell you where to knock. Chapter 5 figures out what's behind each door.

MODULE 05 · ENUMERATION

Service enumeration — what's behind each port

Once nmap shows port 80 open, you don't just stop there — you enumerate every URL, every default page, every share, every banner. Service enumeration is where most pen-tests find their first real foothold.

Port / serviceToolWhat you're looking for
80 / 443 (HTTP/S)gobuster, dirb, niktohidden directories, admin panels, default pages, backup files (config.old, .git).
139 / 445 (SMB)smbclient -L, enum4linux -aanonymously-listable shares, user lists, password policies.
21 / 22 (FTP/SSH)nc -nv, curlbanner versions; anonymous FTP; SSH algorithms.
25 (SMTP)nc + VRFY / EXPNopen relays, valid usernames.
3306 / 5432 (DB)nmap -sV --script=mysql-infoversion, default creds (root/<blank>).
$ gobuster dir -u http://target -w /usr/share/wordlists/dirb/common.txt $ nikto -h http://target $ smbclient -L //10.10.10.10 -N $ enum4linux -a 10.10.10.10 $ nc -nv 10.10.10.10 21

Now you know what's installed and what version. Chapter 6 turns that into a list of vulnerabilities.

MODULE 06 · VULN ID

Vulnerability identification

Enumeration gives you a fingerprint (vsftpd 2.3.4, Apache 2.4.7, OpenSSH 6.6.1p1). Vulnerability identification turns that fingerprint into a CVE list, a CVSS score and a known exploit path.

The local exploit database

Kali ships with the entire Exploit-DB archive, indexed for searchsploit:

$ searchsploit vsftpd 2.3.4 Exploit Title | Path ---------------------------------------------------+----- vsftpd 2.3.4 - Backdoor Command Execution | unix/remote/49757.py

CVSS in 60 seconds

Every finding in your report carries a CVSS Base Score (0.0 - 10.0):

  • 9.0 - 10.0 Critical — remote pre-auth RCE.
  • 7.0 - 8.9 High — auth-required RCE / SQLi extracting hashes.
  • 4.0 - 6.9 Medium — reflected XSS, IDOR.
  • 0.1 - 3.9 Low — verbose error messages, missing security headers.
Free wins. Before launching exploits, check default credentials. admin/admin, root/toor (Kali!), tomcat/tomcat, postgres/postgres account for an embarrassing fraction of real-world breaches.

You have a list of likely weaknesses. Chapter 7 gives you another source of intelligence: the wire itself.

MODULE 07 · SNIFFING

Network sniffing & man-in-the-middle

When you can read traffic on the wire (your own VPN segment, loopback, or an authorized switch span), plaintext credentials and session cookies fall out. tcpdump captures them; tshark / Wireshark dissect them.

StepCommand
Live capture (10 packets, any iface)sudo tcpdump -i any -c 10
Read a saved pcaptshark -r capture.pcap
HTTP onlytshark -r capture.pcap -Y http
Hunt plaintext credsstrings capture.pcap | grep -i pass

ARP poisoning (theory only)

In a switched LAN, ARP poisoning sends forged ARP-reply messages so that the victim's machine and the gateway both believe your MAC is the other party's. All their traffic now flows through you (a man-in-the-middle). arpspoof -i tap0 -t victim -r gateway with echo 1 > /proc/sys/net/ipv4/ip_forward is the eJPT classic. Run this only in a lab you own.

Don't run live arpspoof or telnet sniffing on a real network without written authorization. It is a textbook CFAA / Computer Misuse Act offence.

From the network we move into the most-attacked surface in modern engagements: the web layer. Chapter 8.

MODULE 08 · XSS

Cross-Site Scripting (XSS)

XSS is when attacker-controlled text ends up inside a victim's browser as code. With one crafted link an attacker can read the page, exfiltrate cookies, log keystrokes, or pivot CSRF.

Reflected = single victim per crafted link. Stored = every visitor of the affected page. Stored is the higher-impact bug.

The proof-of-concept that ends every dispute

$ curl 'http://localhost:8080/search?q=<script>alert(1)</script>' <h1>Results for <script>alert(1)</script></h1>

What it achieves & what it can't

Objective

Run JS in someone else's browser. Read the page, exfiltrate cookies, log keystrokes, phish creds, pivot CSRF, deface.

Weakness

HttpOnly cookies + good CSP shut you out. Modern frameworks auto-encode by default, so XSS is rarer in React/Vue front-ends than in 2010-era PHP apps.

Mitigations

Output-encode by context, use auto-escaping templates, enable a strict Content-Security-Policy, set HttpOnly+Secure on session cookies, and sanitise rich-text input with DOMPurify.

XSS lives in the browser. Chapter 9 attacks the database the browser is talking to.

MODULE 09 · SQL INJECTION

SQL Injection

When user input lands inside a SQL string without sanitisation, the attacker can rewrite the query the database actually runs. From a confirmed SQLi you can list databases, dump credential tables, sometimes write files to disk and pivot to RCE.

The four-step exploitation pattern

Confirm with a single quote. Use booleans to map behaviour. UNION to extract columns. Hand off to sqlmap once you've proved it manually.
$ curl "http://localhost:8080/news?id=1'" # 1. confirm $ curl "http://localhost:8080/news?id=1+OR+1=1" # 2. boolean $ curl "http://localhost:8080/news?id=1+UNION+SELECT+1,2,3" # 3. UNION $ sqlmap -u 'http://localhost:8080/news?id=1' --batch --dbs # 4. automate
The fix is one line of code. Use parameterised queries (prepared statements). Stored procs and ORM .where() with placeholders are safe. String concatenation is not.

Manual exploitation with curl is great for proving things. The real workflow uses an HTTP proxy to mutate every request — that's chapter 10.

MODULE 10 · BURP

Burp Suite & HTTP manipulation

Burp Suite is the swiss-army-knife web proxy. It sits between your browser and the target server, captures every request, and lets you mutate any of them by hand. The CLI equivalent is curl -x http://127.0.0.1:8080 … — same idea, less mouse.

Burp's modules at a glance

ModuleUse
ProxyIntercept every request your browser sends.
RepeaterReplay a captured request with edits, instantly.
IntruderFuzz a single parameter with a payload list.
Scanner (Pro)Automated vuln crawl.
Decoderbase64 / URL / hex / hash helpers.
ComparerDiff two responses after small changes.

Curl is your CLI Burp

$ curl -x http://127.0.0.1:8080 http://target # route through Burp $ curl -H 'X-Forwarded-For: 127.0.0.1' http://target/admin # header tampering $ curl -d 'a=b' http://target -X PUT # tamper body / method
SSRF / IDOR. While you have a proxy in front of every request, two extra bug classes are usually one click away: SSRF (server fetches a URL the user supplied — point at http://169.254.169.254/ on AWS for cloud metadata) and IDOR (object IDs aren't authenticated — try /orders/124 → /orders/125).

Web exploitation gets you a foothold on the application. Chapter 11 gets you a foothold on the operating system underneath.

MODULE 11 · METASPLOIT

System exploitation with Metasploit

Metasploit is the largest collection of curated exploits, payloads and post-exploitation modules you can run from one console. The eJPT-style kill chain is: recon → pick exploit → configure → fire → meterpreter → privesc → persistence.

Recon → pick the exploit → configure → fire → meterpreter session → privesc → loot → persistence (only if the ROE allows it).

The console you'll memorise

msf6 > search vsftpd msf6 > use exploit/unix/ftp/vsftpd_234_backdoor msf6 > show options msf6 > set RHOSTS 10.10.10.10 msf6 > set PAYLOAD cmd/unix/interact msf6 > exploit
Image size note. The full metasploit-framework package is >1 GB so this sandbox image leaves it OPT-IN (sudo apt install metasploit-framework). The chapter tasks use searchsploit + the seeded msf.md, meterpreter.md, msfvenom.md cheat sheets.

You're root on the target. Chapter 12 turns that access into a report — and earns the engagement its money.

MODULE 12 · POST-EXPLOITATION

Passwords, privilege escalation & the report

The last chapter wraps three skills: cracking the credentials you've stolen, escalating from a low-priv shell to root, and writing the document that turns all of this into business value.

Password attacks

  • John the Ripper — offline cracker. Loves /usr/share/wordlists/rockyou.txt.
  • Hashcat — GPU-accelerated john. Better for large hash lists.
  • Hydra — online brute-force across SSH, FTP, HTTP-form, RDP, SMB, MSSQL.
  • unshadow — combines /etc/passwd + /etc/shadow into a john-readable file.
$ john --wordlist=/usr/share/wordlists/rockyou.txt hashes.txt $ hydra -L users.txt -P passwords.txt ssh://10.10.10.10 $ unshadow passwd shadow > all.txt && john all.txt

Linux privilege escalation

Once you have a low-privilege shell, the privesc checklist:

$ sudo -l # am I sudoer? $ find / -perm -4000 -type f 2>/dev/null # SUID binaries $ getcap -r / 2>/dev/null # file capabilities $ cat /etc/crontab # scheduled jobs $ ps aux | grep root # root processes

The report

A finding without a clear remediation step is half-finished. Every report has the same skeleton: executive summary → scope → findings → attack narrative → remediation roadmap → appendices → retest section. Each finding gets a CVSS vector, steps to reproduce, evidence and a specific recommendation.

That's the full kill chain. The capstone below ties chapters 4, 5, 9 and 12 together with the seeded vulnerable HTTP server in the sandbox.

CAPSTONE · OPERATION NIGHTHAWK

🎯 Operation NightHawk — your final engagement

Scenario. NightHawk Industries has hired you for a black-box pen-test. They've dropped you on a VPN segment that can reach exactly one host: http://127.0.0.1:8080. You have no credentials, no source code and no documentation. Their CISO wants one thing back: proof you can read the admin flag at /admin/flag. Anything else — their host OS, the public internet, the rest of the lab — is out of scope.

Your kill chain

You've already learnt every move you need. Walk the engagement the same way the chapters did:

  1. Chapter 4 — nmap. What ports are listening? nmap -sV -p 1-9000 127.0.0.1
  2. Chapter 5 — enumerate. Which paths exist? gobuster dir -u http://127.0.0.1:8080 -w /usr/share/wordlists/dirb/common.txt
  3. Chapter 6 — read the index. The home page tells you /admin/flag needs an admin Bearer token.
  4. Chapter 9 — SQL injection. Confirm with a single quote on /news?id=1', then leak the token from the admin_tokens table: id=0+UNION+SELECT+1,token,3+FROM+admin_tokens--
  5. Chapter 10 — header tampering. /admin/flag requires BOTH X-Forwarded-For: 127.0.0.1 AND Authorization: Bearer <token>.
  6. Submit the flag on the form below to close out the engagement and earn the +75 XP bonus.
The full briefing is also seeded inside the desktop at ~/lab/CAPSTONE.md. Open the Pen-test Terminal launcher and run cat ~/lab/CAPSTONE.md.
