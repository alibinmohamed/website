MODULE 01 · FOUNDATIONS

Introduction to Networks

Before anything else, we need a shared mental model of what a network actually is. A computer network is simply a group of devices that can exchange data with one another using an agreed-upon set of rules. That's it. Everything else in this course is detail built on top of that single idea.

The Postal-System Analogy

Think of a network like a postal system. You write a letter (data), put it in an envelope with an address (packet header), drop it in a mailbox (network interface), and the postal service routes it through sorting centres (switches & routers) until it arrives at the destination. If the letter is too big for one envelope, you split it into several numbered envelopes (fragmentation). If an envelope is lost, you might send it again (retransmission). Every networking concept in this course maps onto this analogy.

Why Networks Matter

  • Communication: email, chat, video calls, VoIP.
  • Resource sharing: printers, storage, GPUs, databases.
  • The Internet: the largest network of networks ever built.
  • Security: you cannot defend what you do not understand. Every cyberattack uses a network.

Types of Networks (by Size)

Type Scope Typical Example
PAN — Personal Area Network ~ 1–10 m Your phone + Bluetooth earbuds
LAN — Local Area Network One building / campus Your home Wi-Fi, a UTB computer lab
MAN — Metropolitan Area Network A city A city-wide ISP backbone in Manama
WAN — Wide Area Network Country / continent A multinational bank connecting its branches
Internet Global The public network-of-networks itself

Key Measurements You Will Hear Every Day

  • Bandwidth — the maximum amount of data a link can carry per second (Mbps, Gbps). Think: the width of a pipe.
  • Throughput — the actual data rate you observe. Always ≤ bandwidth.
  • Latency — the time a single packet takes to travel from A to B, measured in milliseconds.
  • Jitter — variation in latency between packets (critical for VoIP).
  • Packet loss — the percentage of packets that never arrive.
Professor's note: Bandwidth and speed are not the same thing. A 1 Gbps link with 200 ms latency will feel slower than a 50 Mbps link with 20 ms latency for most interactive applications.
Lab 1 — Meet Your Own Network
  1. Open a terminal on your own laptop (Terminal on macOS/Linux, cmd or PowerShell on Windows).
  2. Run a round-trip test to Cloudflare and to a public training target:
$ ping -c 4 1.1.1.1 $ ping -c 4 testphp.vulnweb.com # on Windows: ping testphp.vulnweb.com
  1. Record the minimum, average, and maximum round-trip times.
  2. On paper or in a drawing app, sketch your home network: modem, router, Wi-Fi, laptop, phone. Label what type of network each segment is (PAN, LAN, WAN).

MODULE 02 · MENTAL MODEL

The OSI Model

The Open Systems Interconnection (OSI) model is the single most important mental model in networking. It splits the impossibly complex problem of "how do two computers talk?" into seven smaller, independent problems called layers. Each layer only has to do its own job and hand the data off to the layer below.

Why Layering?

Layering is separation of concerns. The engineer who designs a web browser should not have to worry about whether the cable is copper or fibre. The engineer who lays the cable should not have to know HTTP. As long as every layer respects its interface, the layers can evolve independently.

The Seven Layers (Top-Down)

7ApplicationDataHTTP · DNS · SSH · FTP
6PresentationDataTLS · JPEG · ASCII
5SessionDataNetBIOS · RPC
4TransportSegment / DatagramTCP · UDP
3NetworkPacketIP · ICMP · routing
2Data LinkFrameEthernet · MAC · ARP
1PhysicalBitsCables · Wi-Fi radio · voltages

Mnemonic (memorise this)

"All People Seem To Need Data Processing" — Application, Presentation, Session, Transport, Network, Data Link, Physical (top → bottom).

OSI vs TCP/IP

The real Internet runs on the simpler TCP/IP model, which collapses the 7 OSI layers into 4 practical ones: Application, Transport, Internet, and Link. In practice, engineers mix-and-match the vocabulary: you will hear "layer 3" (meaning IP) and "layer 7 firewall" (meaning application-aware) every day of your career.

Encapsulation — The Most Important Idea

When your browser sends an HTTP request, the data travels downwards through the stack. Each layer wraps the data in its own header, like Russian nesting dolls:

[ Ethernet | IP | TCP | HTTP data ] ← what leaves your NIC

On the receiving computer, the same happens in reverse: each layer strips off its header and hands the inner data upwards. That is de-encapsulation.

Lab 2 — Trace a Request Through the Stack
  1. Install Wireshark (free, wireshark.org).
  2. Start a capture on your active network interface.
  3. Visit http://example.com (HTTP, not HTTPS) in your browser.
  4. Stop the capture. Find the first HTTP GET / packet.
  5. Click the packet and expand the tree on the left. You will see, from top to bottom: Frame → Ethernet → IP → TCP → HTTP. Match each expansion to an OSI layer.
Deliverable: a screenshot with your own annotations labelling each layer and its PDU (frame, packet, segment, data).
MODULE 03 · IDENTITY

IP Addresses

If a network is a postal system, the IP address is the street address. It uniquely identifies a network interface so that packets can be delivered to the correct machine anywhere in the world.

IPv4 at a Glance

An IPv4 address is 32 bits, usually written as four decimal octets separated by dots (e.g. 192.168.1.10). Each octet is 8 bits, so each value is in the range 0–255.

Notation Value
Dotted decimal 192.168.1.10
Binary 11000000.10101000.00000001.00001010
Hexadecimal C0.A8.01.0A

Public vs Private (RFC 1918)

Because IPv4 only allows ~4.3 billion addresses, we reserve three ranges for private use inside homes and offices. Your router uses NAT to let many private devices share one public address.

Range CIDR Typical Use
10.0.0.0 – 10.255.255.255 /8 Large enterprises
172.16.0.0 – 172.31.255.255 /12 Medium networks
192.168.0.0 – 192.168.255.255 /16 Home routers

Special Addresses Worth Knowing

  • 127.0.0.1 — loopback. Refers to this machine.
  • 0.0.0.0 — any address. "Listen on everything."
  • 255.255.255.255 — limited broadcast.
  • 169.254.x.x — APIPA. Your OS auto-assigns this when DHCP fails. If you see it, you have no working DHCP.

Subnet Masks & CIDR

A subnet mask tells a device where the network portion of an address ends and the host portion begins. Modern notation uses CIDR: /24 means "the first 24 bits are the network", which equals 255.255.255.0.

IPv6 — The Successor

IPv6 addresses are 128 bits, written as eight groups of four hex digits (e.g. 2001:0db8:85a3:0000:0000:8a2e:0370:7334). That gives 3.4 × 1038 addresses — enough for every grain of sand on Earth to have its own public IP.

Professor's note: Loopback in IPv6 is simply ::1. Any run of zeros can be compressed with ::, but only once per address.
Lab 3 — Know Your Own Address
# macOS / Linux $ ifconfig # or: ip a # Windows > ipconfig /all
  1. Identify your IPv4 address, subnet mask, default gateway, and DNS servers.
  2. Classify your address: public or private? Which RFC 1918 range?
  3. Note your IPv6 address if you have one. Is it link-local (fe80::/10) or global?
  4. Find your MAC address on the same interface.
Deliverable: a filled-in table with IPv4, mask, gateway, DNS, MAC, and IPv6 for your active network interface, plus a one-line classification of each.
MODULE 04 · ENDPOINTS

Ports

An IP address gets a packet to the right machine. But a single machine runs dozens of services at once — a web server, an SSH daemon, a DNS resolver. How does the OS know which program should receive an incoming packet? The answer is the port number.

The Apartment Analogy

The IP address is the building. The port is the apartment number. The combination of (IP address, port) is called a socket and it uniquely identifies one endpoint of a conversation.

The Port Number Space

Ports are 16-bit numbers, so they range from 0 to 65535.

Range Name Meaning
0 – 1023 Well-known Reserved for standard services. Binding usually requires admin rights.
1024 – 49151 Registered Assigned by IANA to specific products.
49152 – 65535 Dynamic / Ephemeral Your OS picks one of these as the source port for outbound connections.

Ports Every Networker Memorises

Port Protocol Notes
20 / 21FTPData / Control — plaintext, legacy.
22SSH / SFTPEncrypted remote shell & file transfer.
23TelnetPlaintext remote shell — do not use.
25SMTPSending mail between servers.
53DNSDomain name resolution.
67 / 68DHCPAutomatic IP configuration.
80HTTPPlaintext web.
110 / 143POP3 / IMAPReceiving mail.
443HTTPSEncrypted web (TLS).
445SMBWindows file sharing.
3389RDPWindows remote desktop.

Source Port vs Destination Port

Every TCP or UDP segment carries two port numbers. When your browser talks to testphp.vulnweb.com:80, the server sees a source port like 54321 (an ephemeral port your OS picked) and a destination port of 443. When the server replies, the source and destination flip. This is how one server can simultaneously serve thousands of clients on port 443.

Security angle: every open port is a potential attack surface. The first thing an attacker does is scan for open ports. The first thing a defender does is close the ones they do not need.
Lab 4 — What's Listening on My Machine?
# List all listening sockets $ netstat -an | grep LISTEN # macOS / Linux $ ss -tulpn # modern Linux > netstat -ano # Windows
  1. List every port your machine is currently listening on.
  2. For each one, identify which program is bound to it (use lsof -i :PORT on macOS/Linux, or the PID column on Windows and tasklist).
  3. Ask yourself: do I actually need each of these services running?
Deliverable: a table of port → program → purpose → keep or close? for at least 5 listening ports on your system.
MODULE 05 · HOW DATA FLOWS

Service Types

Not every conversation on a network works the same way. A video call, a file download, and a DNS lookup have very different requirements. Networking gives us a small number of service types and lets each application pick the right one.

1. Connection-Oriented vs Connectionless

TCP (connection-oriented) UDP (connectionless)
SetupThree-way handshake firstJust send
DeliveryGuaranteed & orderedBest-effort — may lose / reorder
OverheadHigherLower
Good forWeb, SSH, file transfer, emailDNS, VoIP, video, games

The TCP Three-Way Handshake

Before any data is exchanged, TCP performs a three-step "hello" that synchronises both sides on starting sequence numbers. You will see this in Wireshark every day.

Client
Server
— SYN →
← SYN, ACK —
— ACK →

After the third packet, the connection is established and real data can flow in both directions. Tearing down the connection uses a similar exchange of FIN and ACK packets.

2. Delivery Scopes

  • Unicast — one sender → one receiver (the default for almost everything).
  • Broadcast — one sender → every device on the LAN (used by ARP, DHCP).
  • Multicast — one sender → a group of interested receivers (IPTV, routing protocols).
  • Anycast — one sender → the nearest of many identical receivers (how 8.8.8.8 and CDNs work).

3. Service Models

  • Client–Server: one side requests, the other serves (web, email, DNS).
  • Peer-to-Peer: every node is both client and server (BitTorrent, some blockchains).
Lab 5 — Watch TCP and UDP Live
  1. Start a Wireshark capture.
  2. In your browser, visit http://testphp.vulnweb.com. Stop capture.
  3. Filter by tcp.flags.syn == 1. Find the three-way handshake: SYN → SYN/ACK → ACK.
  4. Now filter by udp.port == 53. Inspect a DNS query and its reply — notice there is no handshake, no retransmissions, just a request and a response.
Deliverable: two annotated screenshots — one of a full TCP handshake and one of a UDP DNS exchange — explaining in your own words why each protocol was the right choice.
MODULE 06 · SHAPE

Topology

Topology is the shape of a network — both the physical wiring and the logical data paths. Choosing the right topology affects performance, cost, and how gracefully the network fails.

Physical vs Logical

  • Physical topology — how cables and devices are actually arranged.
  • Logical topology — how data appears to flow between nodes, which can differ from the wiring.

Classical Topologies

Star

Every device connects to a central switch. Today's default. One cable fault affects only one device — but the switch itself is a single point of failure.

Bus

All devices share one backbone cable. Cheap but one cable fault breaks everything. Mostly historical.

Ring

Each device connects to exactly two neighbours forming a closed loop (Token Ring, FDDI). Predictable but fragile.

Mesh

Every node connects to every other. Extremely reliable, extremely expensive. Used in the Internet backbone and some wireless sensor networks.

Tree

Hierarchical — a root switch with branches. Most enterprise LANs are a star-of-stars, which is a tree.

Hybrid

A combination chosen for cost/performance reasons. The real Internet is a hybrid of every topology on this list.

The Devices You'll Find on a Diagram

Device OSI Layer Role
Hub1Dumb repeater. Obsolete — repeats every bit to every port.
Switch2Forwards frames using MAC addresses. Learns a MAC table.
Router3Forwards packets between different networks using IP.
Access Point1–2Bridges Wi-Fi clients onto a wired LAN.
Firewall3–7Allows / denies traffic based on rules.
Modem1–2Translates between ISP's medium (fibre, DSL, cable) and Ethernet.

Collision & Broadcast Domains

A collision domain is a segment where two devices can have their signals overlap. Switches break up collision domains (one per port). A broadcast domain is a segment where a broadcast frame reaches every device. Routers break up broadcast domains.

    Anatomy of an HTTP Request

    GET /index.html HTTP/1.1 Host: testphp.vulnweb.com User-Agent: Mozilla/5.0 Accept: text/html (blank line)

    The server replies with a status line (HTTP/1.1 200 OK), headers, a blank line, and the response body. HTTPS is exactly the same HTTP — just wrapped inside a TLS tunnel after a handshake that proves the server's identity via a certificate.

    DNS in Two Steps

    1. Your resolver asks "what is the A record for testphp.vulnweb.com?"
    2. The authoritative server replies with an IP like 44.228.249.3. Your resolver caches the answer for a TTL (time to live).
    Professor's note: DNS is often called "the phone book of the Internet" — but it's also the most frequently exploited protocol. DNS hijacking, cache poisoning, and tunnelling are all topics you'll revisit in Web Security and Digital Forensics.
    Lab 7 — Plaintext vs Encrypted in the Wild
    $ curl -v http://example.com $ curl -v https://example.com
    1. Run both commands. Compare the verbose output — notice the TLS handshake in the second one.
    2. Start a Wireshark capture and repeat. Filter by http then by tls. Can you read the URL in the HTTP capture? Can you read anything in the TLS capture?
    3. Run nslookup testphp.vulnweb.com and then nslookup -type=MX testphp.vulnweb.com. Record the results.
    Deliverable: a short report explaining, with screenshots, exactly why HTTPS protects privacy and HTTP does not, plus the A and MX records you resolved for testphp.vulnweb.com.
    MODULE 08 · THE ENGINEER'S TOOLBOX

    Command Lines

    A real networking engineer does 90% of their diagnostic work from a terminal. These commands are the stethoscope and blood-pressure cuff of the field — you must be fluent in them.

    The Essential Eight

    Command Purpose Typical Use
    ping Is the host reachable? How fast? ping 8.8.8.8
    traceroute / tracert Which hops does a packet pass through? traceroute testphp.vulnweb.com
    ipconfig / ifconfig / ip a What is my own IP configuration? ip a
    netstat / ss What sockets are open on this machine? ss -tulpn
    nslookup / dig Resolve DNS records manually dig testphp.vulnweb.com ANY
    arp Show / clear the IP→MAC mapping table arp -a
    route / ip route Inspect the routing table ip route
    curl / wget Speak HTTP/HTTPS from the shell curl -I http://testphp.vulnweb.com

    A Structured Way to Troubleshoot

    When something is broken, follow the OSI model from the bottom up:

    1. Layer 1 (Physical) — Is the cable plugged in? Is the Wi-Fi associated?
    2. Layer 2 (Data Link) — arp -a. Can you resolve the MAC of the gateway?
    3. Layer 3 (Network) — ping the gateway, then ping 1.1.1.1 (no DNS needed).
    4. DNS — nslookup testphp.vulnweb.com. If this fails but pinging 1.1.1.1 works, your DNS is broken.
    5. Layer 4+ — curl -v to check if the app-layer service is actually responding.

    Cheat Sheet — "I Cannot Reach the Internet"

    $ ip a # do I have a valid IP? $ ip route # do I have a default gateway? $ ping <gateway> # can I reach my own router? $ ping 1.1.1.1 # can I reach the Internet? $ nslookup example.com # does DNS work? $ curl -v https://example.com # does the app layer respond?
    Lab 8 — The Diagnostic Walkthrough
    1. Starting from a freshly opened terminal, run every command in the cheat sheet above against your own machine and testphp.vulnweb.com.
    2. Record the output of each command.
    3. For traceroute testphp.vulnweb.com, identify the first public hop. That is your ISP's edge router.
    4. Disable your Wi-Fi for 5 seconds and re-run the commands. Observe which ones fail first and why.
    Deliverable: a log file (diagnostics.txt) containing the output of every command, with comments explaining what each result told you.
    MODULE 09 · KEEPING TIME

    Synchronization

    Modern networks are useless if the devices on them disagree about when things happen. Synchronization — of clocks, of connections, and of state — is one of the deepest topics in networking and one of the easiest places for security to break.

    1. Why Time Synchronization Matters

    • Logs: if your firewall and your server disagree by 5 minutes, you can't correlate an attack.
    • TLS certificates: if your clock is wrong, every HTTPS site will look expired or not-yet-valid.
    • Kerberos / Active Directory: refuses authentication if clocks differ by more than 5 minutes — this is the #1 cause of "I can't log in to the domain."
    • Two-factor codes (TOTP): rotate every 30 seconds and fail if the phone and server disagree.
    • Distributed databases & blockchains: rely on ordered, timestamped events.

    2. NTP — Network Time Protocol

    NTP runs on UDP port 123. Devices ask a time source for the current time and adjust their local clock gradually to avoid jumping. Time sources form a hierarchy called strata:

    Stratum Source
    0Atomic clock, GPS receiver — the reference itself.
    1Servers directly connected to a stratum-0 source.
    2Servers that synchronise from stratum 1 over the network.
    …Up to stratum 15. Stratum 16 means "not synchronised".

    Public pools you can use for free: pool.ntp.org, time.google.com, time.apple.com, time.windows.com.

    3. TCP Synchronization — The SYN Flag Revisited

    "Synchronization" in networking doesn't only mean clocks. Every TCP connection begins with a SYN segment, short for synchronize sequence numbers. Both ends pick a random initial sequence number and use the handshake to agree on where the byte stream starts. From that point, every byte sent is numbered, which is how TCP guarantees in-order, reliable delivery.

    Security angle: a SYN flood is a classic denial-of-service attack where the attacker sends thousands of SYN packets and never completes the handshake, exhausting the server's connection table. Modern OSes defend with SYN cookies.

    4. Clock Skew & Drift

    • Skew — the constant offset between two clocks at a single moment.
    • Drift — how fast that skew grows because every crystal oscillator runs at a slightly different rate.
    • This is exactly why we need NTP to continuously correct, not just set-and-forget.

    5. Beyond NTP

    • PTP (IEEE 1588) — Precision Time Protocol. Sub-microsecond accuracy, used in finance, telecom, and broadcast.
    • NTS — Network Time Security. Adds cryptographic authentication to NTP so you can trust the time you receive.
    • Distributed consensus (Paxos, Raft) — how clusters stay "in sync" on state when they can't fully trust any single clock.
    Lab 9 — Is Your Clock Lying?
    # macOS $ sntp -sS time.apple.com # Linux (systemd) $ timedatectl status $ timedatectl timesync-status # Windows (PowerShell, run as admin) > w32tm /query /status > w32tm /stripchart /computer:time.windows.com /samples:5
    1. Run the command for your OS and record your current offset from the reference server. Is your clock ahead or behind?
    2. In Wireshark, capture for 30 seconds while filtering on ntp. You should see your OS periodically asking a time server for the time.
    3. Now capture again while running curl http://testphp.vulnweb.com. Filter by tcp.flags.syn == 1 and tcp.flags.ack == 0 to isolate the initial SYN. Note the sequence number chosen.
    4. Deliberately set your system clock 10 minutes into the past and try visiting any HTTPS site. Note the certificate error. Reset your clock.
    Deliverable: a short write-up that answers: (a) what was your clock's offset? (b) how often does your OS sync with NTP? (c) which error did HTTPS show when your clock was wrong, and what OSI layers did that error involve?

    Capstone — Put It All Together

    You've learned what networks are, the OSI model, IP addresses, ports, service types, topology, application protocols, the command-line toolbox, and synchronization. For your capstone, design a small office network in GNS3, EVE-NG, or Cisco Packet Tracer that uses every concept above:

    • Two subnets (Staff 192.168.10.0/24, Guest 192.168.20.0/24) separated by a router.
    • A switch per subnet (star topology).
    • DHCP on both subnets; an internal DNS server for office.local.
    • A web server on port 443 reachable only from Staff.
    • An NTP server (or upstream pool) so every device agrees on the time.
    • A one-page written report that identifies, for each device, which OSI layer(s) it operates on and which ports you have opened.

    When your report and topology file are ready, share them in the club Discord for feedback from a senior member. Completing this capstone qualifies you to move on to Web Security and Digital Forensics.
